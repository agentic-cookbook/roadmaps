# Plan Bugfix Roadmap

Lightweight roadmap generator for bugfixes. Takes a list of existing GitHub issues and produces a Feature Definition and Feature Roadmap, ready for `/implement-roadmap-interactively` or the `implement-roadmap-agent`.

Unlike `/plan-roadmap`, this skill:
- **Skips the discussion phase** — you already know what the bugs are
- **Uses existing GitHub issues** — no new issues are created (unless you provide descriptions instead of issue numbers)
- **Requires only one approval** — shows the full plan once, then writes everything
- **Groups and orders bugs** by component/area automatically

---

## ABSOLUTE RULE: NO IMPLEMENTATION CODE

Same as `/plan-roadmap` — this skill produces planning documents only. No code, no fixes, no implementation.

---

## Step 0: Validate Environment

```bash
git rev-parse --is-inside-work-tree
```
If this fails, **STOP**.

```bash
gh auth status
```
If this fails, **STOP** — tell the user to run `gh auth login`.

Create directories:
```bash
mkdir -p .claude/Features/FeatureDefinitions
mkdir -p .claude/Features/Active-Roadmaps
mkdir -p .claude/Features/Completed-Roadmaps
mkdir -p .claude/Features/Completed-Features
```

Ensure `.claude/Features/` is tracked by git:
```bash
git check-ignore -q .claude/Features/FeatureDefinitions/test 2>/dev/null && echo "IGNORED" || echo "TRACKED"
```
If `IGNORED`, add negation rules to `.gitignore`, commit and push.

---

## Step 1: Resolve the Bug List

Parse `$ARGUMENTS` to determine which bugs to include.

### If `$ARGUMENTS` is `all` or empty:
Fetch all open issues from the current repo:
```bash
gh issue list --state open --json number,title,body,labels --limit 100
```
Present the list to the user and ask which ones to include, or confirm all.

### If `$ARGUMENTS` is a list of issue numbers (e.g., `3 4 5 6 7 8`):
Fetch each issue:
```bash
gh issue view <number> --json number,title,body,labels
```

### If `$ARGUMENTS` contains text descriptions (not numbers):
These are new bugs without existing issues. They will need issues created in Step 4.

Collect all bugs into a list with: number (if existing), title, body/description, labels.

---

## Step 2: Group and Order

Analyze the bug list and:

1. **Group by component/area** — look at issue titles, labels, and descriptions for common patterns (e.g., `[dashboard]`, `[agent]`, `[plan-roadmap]`)
2. **Order within groups** — dependencies first, then quick wins, then larger items
3. **Assign complexity** — S (< 30 min), M (30 min - 2 hours), L (> 2 hours) based on the description
4. **Identify dependencies** — if fixing bug A requires bug B to be fixed first

---

## Step 3: Generate the Feature Name

Create a PascalCase feature name that describes the batch. Examples:
- `DashboardUIFixes` (if all bugs are dashboard-related)
- `BugfixBatch2026Mar` (if bugs span multiple areas)
- `DashboardV11Fixes` (if versioned)

---

## Step 4: Present the Plan

Show the user the complete plan in one block:

```
=== BUGFIX ROADMAP: <FeatureName> ===

Bugs: <N> total (<X> existing issues, <Y> new)
Groups:
  - <GroupA>: <N> bugs
  - <GroupB>: <N> bugs

Steps:
  1. [S] <title> — #<issue> or NEW
  2. [M] <title> — #<issue> or NEW
  ...

Accept this plan? I will create the Feature Definition, Roadmap,
and any new GitHub issues. (yes/no/edit)
```

**STOP. Wait for the user to approve.**

- If `yes`: proceed to Step 5
- If `edit`: ask what to change, revise, re-present
- If `no`: **STOP**

---

## Step 5: Create New GitHub Issues (if any)

For any bugs that don't have existing issues, create them:

```bash
gh issue create --title "<title>" --body "<description>"
```

Verify each with `gh issue view`.

---

## Step 6: Write Feature Definition

Write `.claude/Features/FeatureDefinitions/<FeatureName>-FeatureDefinition.md`:

```markdown
# Feature Definition: <FeatureName>

**Created**: <today's date>
**Status**: In Progress

## Goal and Purpose

Batch bugfix for <N> issues across <areas>. Generated by /plan-bugfix-roadmap.

## Acceptance Criteria

- All <N> bugs fixed and verified
- All GitHub issues closed
- All PRs merged

## Verification Strategy

| Check | Command / Approach |
|-------|-------------------|
| Build | <project build command, or `true` if none> |
| Test | <project test command, or manual verification> |
```

Commit and push.

---

## Step 7: Write Feature Roadmap

Write `.claude/Features/Active-Roadmaps/<FeatureName>-FeatureRoadmap.md` with one step per bug.

Each step should have:
- `### Step N: <bug title>` — the description from the issue title
- `**Description**: <summary from issue body>`
- `**GitHub Issue**: <repo>#<number>`
- `**Status**: Not Started`
- `**Complexity**: S | M | L`
- `**Dependencies**: None | Step N`
- `**Acceptance Criteria**:` — derived from the issue body
- `**PR**: _TBD_`

Set:
- `**Status**: Not Started`
- `**Phase**: Ready`

Commit and push.

---

## Step 8: Final Summary

```
=== BUGFIX ROADMAP COMPLETE: <FeatureName> ===

Feature Definition: .claude/Features/FeatureDefinitions/<FeatureName>-FeatureDefinition.md
Roadmap: .claude/Features/Active-Roadmaps/<FeatureName>-FeatureRoadmap.md
Steps: <N>
Issues: #<first> through #<last>
Phase: Ready

To implement, run:
  /implement-roadmap-interactively
  or: claude --agent implement-roadmap-agent "Implement <FeatureName>"
```
