01System Overview

Cat Herding turns a conversation about what you want to build into shipped code through structured planning, deterministic step execution, and real-time progress tracking. The LLM handles individual tasks; a Python coordinator decides what to do next.

Core Principles architecture
  • Deterministic step selection - A Python script picks the next step via regex parsing. The LLM never decides what to do next.
  • File-based records as source of truth - YAML frontmatter Markdown files in ~/.roadmaps/ track all state. No database required for the core workflow.
  • Decoupled dashboard - The Flask dashboard service knows nothing about git, files, or Claude Code. Agents push data to it via REST.
  • Worktree isolation - Code goes in the git worktree. Roadmap files stay in ~/.roadmaps/. Never mix them.
  • Draft PR first - A draft PR is created at the start. Each step pushes and comments on the PR. The final step marks it ready, waits for CI, and merges.

02Roadmap System View on GitHub →

The roadmap tools automate the full roadmap lifecycle — from conversation to shipped code.

/plan-roadmap — Two-Phase Feature Planning planning

Collaborative planning workflow that turns a feature idea into a structured roadmap through conversation, then structured artifact creation.

PhaseAction
Phase 1 — DiscussionConversational exploration of the feature idea. Clarifies scope, constraints, and acceptance criteria. No files created.
Phase 2 — PlanningProduces a Feature Roadmap file in ~/.roadmaps/ with goal, scope, acceptance criteria, and implementation steps.
Alignment ReviewSix-point intent verification ensures the roadmap matches the original discussion before finalizing.

Output: a single Roadmap.md file with YAML frontmatter. GitHub issues use {{REPO}}#{{ISSUE_NUMBER}} placeholders — issues are created by /implement-roadmap.

Guard: no implementation code is ever generated. Only Markdown files in ~/.roadmaps/.

/implement-roadmap — Automated Step Execution implementation

Deterministic coordinator that implements a planned roadmap step-by-step, using a Python script for step selection and the Agent tool for each worker.

ComponentRole
Python coordinatorDeterministic step selection via regex parsing — the LLM never decides what to do next
Worker agentImplements one step: code, build, test, commit, push, comment on PR
Dashboard syncReal-time progress updates pushed to the Flask dashboard via REST API
Draft PR flowCreates a draft PR at the start; each step comments on it; final step marks it ready and merges

Worktree isolation: code goes in a git worktree. Roadmap files stay in ~/.roadmaps/. Never mixed.

Review agents: configurable per-step and final review agents (code-reviewer, silent-failure-hunter, pr-test-analyzer, etc.).

/repair-roadmap — Re-Plan Incomplete Roadmaps planning

Re-plans an existing incomplete roadmap while preserving its identity. Archives the original for comparison.

StepAction
ResolveFinds the roadmap to repair using the coordinator scanner
DiagnoseReads current roadmap state, identifies stuck or incorrect steps
Re-planProduces a revised roadmap using the same planning logic as /plan-roadmap
ArchiveRenames the original roadmap for comparison; new roadmap keeps the same ID, project, and github-user

NO SCOPE REDUCTION rule: the repaired roadmap must not lose any functionality from the original. Step count comparison enforces this.

Repair log: every action is logged to repair.log inside the roadmap directory.

/plan-bugfix-roadmap — Batch Bug Planning planning

Lightweight roadmap generator for batches of existing GitHub issues. Skips discussion and produces a ready-to-implement roadmap.

FeatureDetail
InputIssue numbers (e.g., 3 4 5 6 7 8) or all to fetch all open issues
GroupingAutomatically groups and orders bugs by component/area
ApprovalSingle approval — shows the full plan once, then writes everything
OutputFeature Roadmap in ~/.roadmaps/, ready for /implement-roadmap

Unlike /plan-roadmap: no discussion phase, uses existing GitHub issues (no new issues created), requires only one approval.

/progress-dashboard — Live Progress UI dashboard

Reusable live progress dashboard with a Flask REST API backend and web UI frontend. Any agent or skill can push updates.

ComponentDetail
ServerFlask REST API on port 8888, SQLite (WAL mode) database, SSE/polling for real-time updates

Worker agent that implements exactly one roadmap step. Launched by the coordinator via the Agent tool, returns when done.

02Code Generation and Review Guidelines View on GitHub →

Shared coding guidelines installed globally at ~/.claude/guidelines/. Used for both code generation and code review. Every Claude Code session reads these automatically. All rules use hierarchical CG- numbering - CG-X.Y (file.section) or CG-X.Y.Z (file.section.subrule) - for unambiguous cross-referencing.

CG-1 - General Coding Guidelines (21 rules) [source] all projects
  • CG-1.1 Prefer native controls - Always use platform built-in frameworks first (Swift Concurrency over raw threads, Room/SwiftData over raw SQLite). Explicitly note which native controls are used and why.
  • CG-1.2 Prefer proven open-source - When no native solution exists, research battle-tested libraries and present options before building custom. Custom is a deliberate choice, not a default.
  • CG-1.3 Surface all design decisions - Any LLM choice affecting behavior must be noted and approved. Record in a Design Decisions section so all platform implementations stay consistent.
  • CG-1.4 No blocking the main thread - Use platform async primitives: Swift Concurrency (async/await, actors), Kotlin Coroutines (Dispatchers.IO), Promise/async (Web), asyncio (Python), async/await + DispatcherQueue (.NET).
  • CG-1.5 Always show progress - Determinate progress bar when total is known. Spinner/skeleton/shimmer when not. Never a frozen UI.
  • CG-1.6 Comprehensive unit testing - Prioritize unit tests over integration. Test state transitions, edge cases, serialization. Every implementation needs a test file.
  • CG-1.7 Small atomic commits - One logical change per commit. May touch multiple files if part of the same concept. Commit as work progresses.
  • CG-1.8 Post-generation verification - Build for all targets, run tests, lint, verify log output, accessibility audit (tap targets, contrast), code review against best practices. Fix before considering done.
  • CG-1.9 Instrumented logging - Every component instrumented with structured debug logging via platform logger (os.log, Timber, console, logging, ILogger<T>). Log state transitions, async task lifecycle, branching logic.
  • CG-1.10 Deep linking - All views reachable via URL. Apple: Universal Links + onOpenURL. Android: App Links + intent filters. Web: URL routing.
  • CG-1.11 Scriptable and automatable - Apple: AppIntents for Shortcuts. Android: AppActions. Web: API endpoints or query params.
  • CG-1.12 Accessibility from day one - Semantic roles/labels, VoiceOver/TalkBack/screen reader support, keyboard nav, Dynamic Type, WCAG AA contrast (4.5:1 text, 3:1 large), meaningful focus order.
  • CG-1.13 Localizability - No hardcoded strings. Apple: String(localized:). Android: strings.xml. Web: react-intl/i18next.
  • CG-1.14 RTL layout support - Leading/trailing not left/right. Mirror directional icons. Test with RTL locale.
  • CG-1.15 Respect accessibility display options - Respond to reduced motion, high contrast, color inversion, bold text, grayscale. See platform files for full environment key lists.
  • CG-1.16 Privacy and security - Data minimization, opt-in consent, platform secure storage (Keychain/EncryptedSharedPreferences/DPAPI/HttpOnly cookies), no PII logging, TLS only, input sanitization.
  • CG-1.17 Feature flags - All features gated from day one. FeatureFlagProvider interface with local default (UserDefaults/SharedPreferences/localStorage). Backend swap via DI.
  • CG-1.18 Analytics - All user actions instrumented via AnalyticsProvider interface. Logging-only default. No direct backend coupling.
  • CG-1.19 A/B testing - ExperimentProvider interface with local default and debug panel override.
  • CG-1.20 Debug mode - Debug-only panel (not in release): flag overrides, analytics log, A/B picker, env info. Shake (iOS/Android), debug menu (macOS), /debug (web).
  • CG-1.21 Linting from day one - SwiftLint + swift-format, ktlint, ESLint + Prettier, Roslyn Analyzers + dotnet format. Config committed. Runs in build or pre-commit. Auto-fixable.
CG-2 - Engineering Principles (16 rules) [source] all projects
  • CG-2.1 Simplicity - Simple ≠ easy. No interleaving of concerns. Complexity compounds — resist at introduction time.
  • CG-2.2 Make It Work, Make It Right, Make It Fast - Working solution → refactor for clarity → optimize only what measurement proves slow. Never skip phase 2.
  • CG-2.3 Composition over inheritance - Default to composing behaviors from small pieces. Protocols/interfaces over base classes. Wrap rather than subclass.
  • CG-2.4 Dependency injection - Receive dependencies from outside, never construct internally. Protocol types, not concrete types. No service locator.
  • CG-2.5 Immutability by default - let/val/const by default. Value types over reference types. Contain mutation behind clear boundaries (actors, StateFlow, useState).
  • CG-2.6 Fail fast - Detect invalid state at origin. Assertions in debug, typed errors over swallowed exceptions, never empty catch blocks.
  • CG-2.7 Idempotency - Safe to repeat without duplicate effects. Debounce buttons, idempotency keys for APIs, migrations safe to re-run.
  • CG-2.8 Design for deletion - Code is a liability. Easy to throw away without affecting the system. Duplicate rather than couple. Don't abstract prematurely.
  • CG-2.9 YAGNI - Build for today's requirements. Speculative generality costs more to maintain than adding it later.
  • CG-2.10 Explicit over implicit - Visible dependencies, clear naming, parameter passing over ambient state.
  • CG-2.11 Small reversible decisions - Cheap to reverse → decide fast. Expensive to reverse → invest in understanding. Defer binding until last responsible moment.
  • CG-2.12 Tight feedback loops - Fast tests, frequent deploys, early user feedback, automate commit-to-production.
  • CG-2.13 Separation of concerns - One reason to change. If describing a module requires "and," split it.
  • CG-2.14 Least astonishment - Behavior matches expectations. Side effects obvious from API signature.
  • CG-2.15 Manage complexity through boundaries - Ports (interfaces) + adapters (implementations). Test core without databases, UIs, or networks.

Meta-principle: Optimize for Change. Every principle above makes future change cheaper and safer.

CG-12 - Spec Writing Format [source] specs

Format for behavioral specifications (from litterbox and temporal-platform).

  • Frontmatter: version (semver), status (draft/review/accepted/deprecated), created, last-updated, author, platforms, tags, dependencies with version pins.
  • RFC 2119 keywords: MUST, MUST NOT, SHOULD, SHOULD NOT, MAY for all behavioral requirements.
  • REQ-NNN numbering: Sequential, never reused. Every MUST needs at least one test vector.
  • Template variables: {{app_name}}, {{org_package}}, {{api_base_url}}, {{bundle_id}} — resolved at implementation time.
  • 18 standard sections: Overview, Terminology, Requirements, Data Structures, Appearance, States, API Contract, Accessibility, Test Vectors, Edge Cases, Logging, Deep Linking, Privacy, Feature Flags, Analytics, Platform Notes, Design Decisions, Changelog.
  • Test vectors: Behavioral tables (ID | Requirements | Input | Expected) or data JSON blocks. May also be published as separate JSON in vectors/.
  • Logging section: Exact log messages per event. Enables grepping output for verification instead of visual inspection.

Design

CG-9 - UI Design Guidelines (12 principles) [source] all projects

Platform HIG always takes precedence. These defaults fill gaps or establish a cross-platform baseline. References: Apple HIG, Material Design 3, Fluent 2, WCAG 2.1.

  • Visual hierarchy - One primary action per screen. Use size and weight (not just color) to establish importance. Group with proximity, separate with whitespace.
  • Spacing - 4px base unit, 8px primary grid. Scale: 4, 8, 12, 16, 24, 32, 48, 64. No arbitrary values.
  • Typography - Platform system font. Body: 14-17pt (16px cross-platform default). Line height: 1.4-1.5x. 2-3 weights max per screen. 45-75 char paragraph width.
  • Color - Semantic tokens over hex. 1 accent + neutrals + semantic (success/warning/error). Never color alone for meaning. WCAG AA: 4.5:1 text, 3:1 large text & UI components.
  • Layout - Single-column by default. Content-first. Consistent alignment. Platform adaptive layout (Size Classes, CSS media queries, VisualStateManager). One scroll direction per view.
  • State design - Every data view handles four states: Loading (skeleton/spinner), Empty (icon + message + CTA), Error (problem + reason + recovery action), Loaded. Design empty/error states with equal care.
  • Form design - Single-column. Validate on blur, not keystroke. Errors inline below field with icon + color + text. Specific messages ("8+ characters" not "Invalid"). Mark optional fields, not required.
  • Feedback - Inline for field-level, toast/snackbar for non-critical confirmations (3-5s auto-dismiss), dialog only for destructive actions. Never dialog for success. Default focus on Cancel for destructive confirms.
  • Touch targets - Platform HIG first (44pt Apple, 48dp Material, 40epx Fluent, 24px WCAG AA). Cross-platform default: 44x44. 8px spacing between adjacent targets.
  • Animation - Purposeful only. State changes: 100-200ms. Enter/exit: 200-350ms. Page transitions: 300-500ms. Platform easing curves. Always respect reduced-motion.
  • Iconography - Platform icon set first (SF Symbols, Material Symbols, Segoe Fluent Icons). Icons supplement text, not replace it. Consistent size/weight. State icons need color AND shape.
  • Data display - Lists for sequential, Tables for comparable multi-attribute (sortable), Cards for heterogeneous, Grids for uniform visual. Sort/filter at 10+ items, search at 50+.

Client-Server

CG-10 - Networking Guidelines (9 topics) [source] all projects

References: Microsoft REST Guidelines, Google API Design, Zalando Guidelines, RFC 9457.

  • API design - REST with lowercase-hyphen URLs, plural nouns, shallow nesting (max 2). HTTP methods as verbs. URL path versioning (/v1/). Consensus across Microsoft, Google, Zalando.
  • Error responses - RFC 9457 Problem Details (application/problem+json): type, title, status, detail, instance. Extension fields for validation errors.
  • Pagination - Cursor pagination preferred (stable under mutation, consistent perf). Offset for static data with page numbers. Response envelope with data + pagination.
  • Retry & resilience - Exponential backoff with full jitter. Base 1s, max 30s, 3-5 retries. Retryable: 408, 429, 502, 503, 504. Never retry 400/401/403/404. Circuit breaker for cascading failures.
  • Timeouts - Connection: 10s. Read: 30s. Total: 60-120s. Never infinite. Use 202 Accepted + polling for long operations.
  • Caching - HTTP headers (Cache-Control, ETag, If-None-Match). Immutable assets: 1 year. Dynamic: short max-age or no-cache. Sensitive: no-store. Invalidate after mutations.
  • Offline - Optimistic updates + queue-based sync + server-wins as default pattern. ETags for conflict detection. Never silently discard user work.
  • Rate limiting - Respect Retry-After. Track RateLimit-Remaining proactively. Queue and batch rather than fire-and-retry.
  • Real-time - SSE for server-push (80% of cases). WebSocket only for bidirectional. Polling as low-frequency fallback.
CG-11 - Security Guidelines (10 topics) [source] all projects

Supplements general.md rule 16. References: OWASP Top 10, OWASP Cheat Sheets, Mozilla TLS.

  • Authentication - OAuth 2.0 + PKCE for all public clients. System browser for native (never WebView). BFF pattern for SPAs. Short-lived access tokens (5-15 min), refresh token rotation.
  • Token handling - Platform secure storage (Keychain, EncryptedSharedPreferences, DPAPI, HttpOnly cookies). Never localStorage. Never tokens in URLs. Proactive refresh at 75% TTL.
  • Authorization - Server-side only. Deny by default. Least privilege. RBAC with fine-grained permissions. BOLA is #1 API risk - always verify resource ownership.
  • Transport - TLS 1.2 minimum (prefer 1.3). HSTS with preload. Certificate pinning: intermediate CA only, mobile only, backup pins required. AEAD ciphers.
  • CORS - Explicit origin allowlist (never reflect Origin). Never wildcard with credentials. Cache preflights. Common misconfigs: unanchored regex, null origin.
  • CSP - default-src 'none' + nonce-based scripts + strict-dynamic. Never unsafe-inline/unsafe-eval. Report-only mode first. frame-ancestors 'self'.
  • Input validation - Never trust client input. Allowlists over denylists. Validate → sanitize → escape. Parameterized queries only. Context-specific output encoding.
  • Sensitive data - Data minimization in APIs. PII classification by tier. Field-level encryption via KMS. No PII in logs. No internals in error responses. no-store for sensitive responses.
  • Dependency security - Lockfiles mandatory (committed). Automated scanning in CI. Pin versions. SRI for CDN assets. Watch for typosquatting and dependency confusion.
  • Security headers - HSTS, CSP, X-Content-Type-Options: nosniff, X-Frame-Options: DENY, Referrer-Policy, Permissions-Policy.

Review Flow - Per-Step + Final algorithm

Per-Step Review (Step 4.5)

Runs after build and test pass, before pushing. Quick review to catch issues before they accumulate.

  • Default agent: code-reviewer
  • Gets the diff via git diff HEAD~1
  • Receives general.md + platform-specific guideline as context
  • High/critical findings → fix, re-run build+test+review (max 3 iterations)
  • Medium/low → continue (logged for follow-up)

Final Review (Finalize & Merge PR)

Full multi-agent review on the complete PR diff. All configured agents run in parallel.

  • Default agents: code-reviewer + silent-failure-hunter

  • Receives general.md + engineering-principles.md + platform guideline
  • Findings aggregated across all agents
  • High/critical → fix, re-run only failing reviewers (max 3 iterations)
  • Medium → fix if quick, otherwise log as follow-up

04Data Integrity

Guardrails that prevent data loss during planning, repair, and implementation. Born from an incident where a repair agent silently dropped 6 L-complexity steps.

The Incident - Why These Rules Exist algorithm

A roadmap repair ran on CatnipIDEEnhancements (20 steps, 6 L-complexity). The repair agent inherited "reviewable in one sitting" guidance and dropped all L-complexity steps. Result: 14 steps, 6 features silently deleted.

Root causes: no scope reduction rule, no step-count comparison, complexity treated as a filter.

NO SCOPE REDUCTION Rule algorithm
  • Do NOT drop steps because of complexity (L is valid)
  • Do NOT merge steps to "simplify" unless user explicitly asks
  • Do NOT filter by complexity, size, or effort
  • "Fix the large steps" means fix them IN PLACE
  • Complexity (S/M/L) is a descriptor, not a filter
Step-Count Comparison algorithm

Before approval, repair prints: Original N steps (S:x M:x L:x) vs Revised N steps. If any removed, prints WARNING with each removed step name and requires explicit confirmation.

Alignment Review (plan-roadmap v12) algorithm

Six-point check comparing draft Roadmap against Discussion Summary: goal match, requirements coverage, no scope creep, constraints honored, user quotes, nothing skipped. Hard stop on failure - draft discarded, planning restarts.

Implementation Phase - /implement-roadmap v26 algorithm

Startup Sequence

  • Step 1: Resolve roadmap (coordinator scans ~/.roadmaps/ + Roadmaps/, filters by project field)
  • Step 1a: Check for stale artifacts (worktree, branch, PR, Implementing state) - prompt user immediately
  • Step 1b: Initialize implementation log
  • Step 1c: Mark as Implementing (only after cleanup)
  • Step 2: Start dashboard (auto-starts if not running)
  • Step 2b: Create feature branch and worktree (code only, no roadmap files)

Implementation Loop

  • 3a: Coordinator picks next step (deterministic, regex-based)
  • 3b: Print step info
  • 3c: Update dashboard (begin-step)
  • 3d: Launch worker agent via Agent tool
  • 3e: Verify completion, log commit, update dashboard (finish-step)
  • Repeat until done

Step 1: Create Draft PR

The first step pushes the branch and creates a draft PR titled Roadmap: <Name>. The PR stays draft throughout implementation.

Standard Steps: Implement, Build, Test, Push, Comment

Each step: implement code, build, test, update roadmap status in ~/.roadmaps/ (no git commit), push to branch, comment on the draft PR.

Final Step: Finalize & Merge PR

Populates Change History (commits table + PR link), copies <Name>-Roadmap.md flat file into repo, marks PR ready, waits for CI, reviews, merges, cleans up worktree.

Critical Invariant

The worktree is for CODE ONLY. Roadmap files stay in ~/.roadmaps/. The only roadmap artifact that enters the repo is the final flat file copy.

Worker Agent - implement-step-agent v11 algorithm

Implements exactly one step. Receives step number, description, complexity, roadmap path, worktree path, dashboard info, and log path in its prompt.

Three Flows

  • "Create Draft PR": Push branch, create draft PR, register PR on dashboard via pr-created
  • "Finalize & Merge PR": Populate Change History, copy flat roadmap file, mark PR ready, wait for CI, review, merge, cleanup
  • Standard: Read codebase, implement, build, test, update roadmap in ~/.roadmaps/, push, comment on PR

Rules

  • Implement ONLY the specified step
  • Never skip a step
  • Use git -C <path> instead of cd && git
  • Roadmap files live in ~/.roadmaps/, NOT in the worktree
  • On failure, log the error and return (don't retry silently)

06Coordinator Algorithm

The coordinator is a deterministic Python script. It reads roadmap files with regex, decides the next step, and outputs JSON. No LLM involved in step selection. [source]

next-step - Find the next implementable step algorithm

Iterates all ### Step N: sections in order. For each:

  • If status is Complete - skip
  • If type is Manual - add to manual_skipped, skip
  • Otherwise - this is the next step. Return it with the skipped manual steps.

If no implementable step remains, return {"action": "done"}.

for step in steps:
    if step.status == "Complete": continue
    if step.type == "Manual":
        manual_skipped.append(step)
        continue
    return {"action": "implement", "step": step, "manual_skipped": manual_skipped}
return {"action": "done", "manual_skipped": manual_skipped}

Key design decision: Manual steps are skipped, not blocked. The coordinator reports them so the skill can create GitHub issues and mark them on the dashboard.

resolve - Find and select a roadmap algorithm

Scans two locations:

  • ~/.roadmaps/<project>/ - working directory (drafts + in-progress)
  • Roadmaps/ in the repo - committed roadmaps

Project filtering: reads the project frontmatter field from each Roadmap.md. If it doesn't match the current git repo name, the roadmap is excluded.

Implementable check: only roadmaps in Ready or Implementing state are offered. Implementing is included so failed runs can be retried.

Returns single match as {"path": ...}, multiple as {"choose": [...]}, or {"error": ...}.

summary - Progress report algorithm

Parses all steps and returns counts:

{"feature": "AuthMiddleware", "total": 8, "complete": 5,
 "steps": [{"step": 1, "description": "...", "status": "Complete"}, ...]}

07State Machine

State is tracked via dated files in State/. The newest file (by date, then lifecycle order) determines the current state. [source]

Created → Planning → Ready → Implementing → Complete ↓ ↓ Declined Paused → Archived
Lifecycle ordering - Why not alphabetical? algorithm

State files are named YYYY-MM-DD-StateName.md. Sorting alphabetically puts Complete before Ready (C < R), which caused a bug where completed roadmaps showed as Ready.

Fix: sort by date first, then by lifecycle order:

_STATE_ORDER = {
    "Created": 0, "Planning": 1, "Ready": 2,
    "Implementing": 3, "Paused": 4,
    "Complete": 5, "Declined": 6, "Archived": 7,
}

Regression test: test_complete_beats_ready_on_same_date ensures Complete wins over Ready when both share the same date.

State predicates algorithm
  • is_active() - True for Created, Planning, Ready, Implementing, Paused
  • is_implementable() - True for Ready or Implementing (Implementing allows retrying failed runs)
  • current_state() - Returns the state name from the newest State/ file
