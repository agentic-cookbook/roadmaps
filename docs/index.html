01System Overview

Cat Herding turns a feature conversation into shipped code through structured planning, deterministic step execution, and real-time progress tracking. The LLM handles individual tasks; a Python coordinator decides what to do next.

Core Principles architecture
  • Deterministic step selection — A Python script picks the next step via regex parsing. The LLM never decides what to do next.
  • File-based records as source of truth — YAML frontmatter Markdown files in ~/.roadmaps/ track all state. No database required for the core workflow.
  • Decoupled dashboard — The Flask dashboard service knows nothing about git, files, or Claude Code. Agents push data to it via REST.
  • Worktree isolation — Code goes in the git worktree. Roadmap files stay in ~/.roadmaps/. Never mix them.
  • Draft PR first — A draft PR is created at the start. Each step pushes and comments on the PR. The final step marks it ready, waits for CI, and merges.

02Code Generation Guidelines View on GitHub →

Shared coding guidelines installed globally at ~/.claude/guidelines/. Every Claude Code session reads these automatically.

General Coding Guidelines — 21 universal rules [source] all projects
  • 1. Prefer native controls — Always use platform built-in frameworks first (Swift Concurrency over raw threads, Room/SwiftData over raw SQLite). Explicitly note which native controls are used and why.
  • 2. Prefer proven open-source — When no native solution exists, research battle-tested libraries and present options before building custom. Custom is a deliberate choice, not a default.
  • 3. Surface all design decisions — Any LLM choice affecting behavior must be noted and approved. Record in a Design Decisions section so all platform implementations stay consistent.
  • 4. No blocking the main thread — Use platform async primitives: Swift Concurrency (async/await, actors), Kotlin Coroutines (Dispatchers.IO), Promise/async (Web), asyncio (Python).
  • 5. Always show progress — Determinate progress bar when total is known. Spinner/skeleton/shimmer when not. Never a frozen UI.
  • 6. Comprehensive unit testing — Prioritize unit tests over integration. Test state transitions, edge cases, serialization. Every implementation needs a test file.
  • 7. Small atomic commits — One logical change per commit. May touch multiple files if part of the same concept. Commit as work progresses.
  • 8. Post-generation verification — Build for all targets, run tests, lint, verify log output, accessibility audit (tap targets, contrast), code review against best practices. Fix before considering done.
  • 9. Instrumented logging — Every component instrumented with structured debug logging via platform logger (os.log, Timber, console, logging). Log state transitions, async task lifecycle, branching logic.
  • 10. Deep linking — All views reachable via URL. Apple: Universal Links + onOpenURL. Android: App Links + intent filters. Web: URL routing.
  • 11. Scriptable and automatable — Apple: AppIntents for Shortcuts. Android: AppActions. Web: API endpoints or query params.
  • 12. Accessibility from day one — Semantic roles/labels, VoiceOver/TalkBack/screen reader support, keyboard nav, Dynamic Type, WCAG AA contrast (4.5:1 text, 3:1 large), meaningful focus order.
  • 13. Localizability — No hardcoded strings. Apple: String(localized:). Android: strings.xml. Web: react-intl/i18next.
  • 14. RTL layout support — Leading/trailing not left/right. Mirror directional icons. Test with RTL locale.
  • 15. Respect accessibility display options — Respond to reduced motion, high contrast, color inversion, bold text, grayscale. See platform files for full environment key lists.
  • 16. Privacy and security — Data minimization, opt-in consent, platform secure storage (Keychain/EncryptedSharedPreferences/HttpOnly cookies), no PII logging, TLS only, input sanitization.
  • 17. Feature flags — All features gated from day one. FeatureFlagProvider interface with local default (UserDefaults/SharedPreferences/localStorage). Backend swap via DI.
  • 18. Analytics — All user actions instrumented via AnalyticsProvider interface. Logging-only default. No direct backend coupling.
  • 19. A/B testing — ExperimentProvider interface with local default and debug panel override.
  • 20. Debug mode — Debug-only panel (not in release): flag overrides, analytics log, A/B picker, env info. Shake (iOS/Android), debug menu (macOS), /debug (web).
  • 21. Linting from day one — SwiftLint + swift-format, ktlint, ESLint + Prettier. Config committed. Runs in build or pre-commit. Auto-fixable.
Engineering Principles — 15 core principles [source] all projects
  • 1. Simplicity — Simple ≠ easy. No interleaving of concerns. Complexity compounds — resist at introduction time.
  • 2. Make It Work, Make It Right, Make It Fast — Working solution → refactor for clarity → optimize only what measurement proves slow. Never skip phase 2.
  • 3. Composition over inheritance — Default to composing behaviors from small pieces. Protocols/interfaces over base classes. Wrap rather than subclass.
  • 4. Dependency injection — Receive dependencies from outside, never construct internally. Protocol types, not concrete types. No service locator.
  • 5. Immutability by default — let/val/const by default. Value types over reference types. Contain mutation behind clear boundaries (actors, StateFlow, useState).
  • 6. Fail fast — Detect invalid state at origin. Assertions in debug, typed errors over swallowed exceptions, never empty catch blocks.
  • 7. Idempotency — Safe to repeat without duplicate effects. Debounce buttons, idempotency keys for APIs, migrations safe to re-run.
  • 8. Design for deletion — Code is a liability. Easy to throw away without affecting the system. Duplicate rather than couple. Don't abstract prematurely.
  • 9. YAGNI — Build for today's requirements. Speculative generality costs more to maintain than adding it later.
  • 10. Explicit over implicit — Visible dependencies, clear naming, parameter passing over ambient state.
  • 11. Small reversible decisions — Cheap to reverse → decide fast. Expensive to reverse → invest in understanding. Defer binding until last responsible moment.
  • 12. Tight feedback loops — Fast tests, frequent deploys, early user feedback, automate commit-to-production.
  • 13. Separation of concerns — One reason to change. If describing a module requires "and," split it.
  • 14. Least astonishment — Behavior matches expectations. Side effects obvious from API signature.
  • 15. Manage complexity through boundaries — Ports (interfaces) + adapters (implementations). Test core without databases, UIs, or networks.

Meta-principle: Optimize for Change. Every principle above makes future change cheaper and safer.

Python Conventions [source] cat-herding
  • 1. No external deps in core — roadmap_lib uses stdlib only. No PyYAML, no requests.
  • 2. Testing — pytest. Every change needs tests. Every fix needs a regression test. Never touch production DB (port 8888) — use demo port 9888.
  • 3. Type hints — Welcome but not required. Python 3.9 compat.
  • 4. File paths — pathlib.Path, not os.path.
  • 5. YAML frontmatter — Use roadmap_lib.parse_frontmatter(). No PyYAML dependency.
  • 6. Web services — Flask for REST APIs.
  • 7. Database — SQLite with WAL mode via stdlib sqlite3. No ORM.
  • 8. Use roadmap_lib — Don't reimplement. Use existing functions for state, parsing, discovery.
  • 9. Deterministic IDs — From YAML frontmatter. Never random UUIDs.
  • 10. Dashboard is display-only — No git/file knowledge. Agents sync data to it.
  • 11. Shell scripts — main() calls functions only, no inline logic.
  • 12. Logging — logging module with getLogger(__name__).
Spec Writing Format [source] specs

Format for behavioral specifications (from litterbox and temporal-platform).

  • Frontmatter: version (semver), status (draft/review/accepted/deprecated), created, last-updated, author, platforms, tags, dependencies with version pins.
  • RFC 2119 keywords: MUST, MUST NOT, SHOULD, SHOULD NOT, MAY for all behavioral requirements.
  • REQ-NNN numbering: Sequential, never reused. Every MUST needs at least one test vector.
  • Template variables: {{app_name}}, {{org_package}}, {{api_base_url}}, {{bundle_id}} — resolved at implementation time.
  • 18 standard sections: Overview, Terminology, Requirements, Data Structures, Appearance, States, API Contract, Accessibility, Test Vectors, Edge Cases, Logging, Deep Linking, Privacy, Feature Flags, Analytics, Platform Notes, Design Decisions, Changelog.
  • Test vectors: Behavioral tables (ID | Requirements | Input | Expected) or data JSON blocks. May also be published as separate JSON in vectors/.
  • Logging section: Exact log messages per event. Enables grepping output for verification instead of visual inspection.

Planning Phase — /plan-roadmap v11 algorithm

Two sub-phases: Discussion (conversational, no files) and Planning (structured, produces Roadmap.md).

Discussion (Steps 0-3)

  • Validate git repo, create drafts directory, initialize planning log
  • Ask the user what they want to build
  • Natural conversation — understand scope, clarify unknowns
  • Summarize, propose a PascalCase feature name, request transition permission

Planning (Steps 4-6)

  • Enter plan mode with deep thinking
  • Draft Roadmap.md from template — feature definition sections at top, implementation steps below
  • CLI readiness checks: before marking any step Manual, test if the tool is installed and authenticated. Ask user to install/auth or confirm Manual.
  • Present full draft, iterate until approved
  • Write to ~/.roadmaps/<project>/YYYY-MM-DD-<Name>/
  • Create state files: Created, Planning, Ready

Frontmatter Fields

id: "uuid"
project: "repo-name"
github-user: "username"
plan-version: "11"
created: "YYYY-MM-DD"
description: "one-line summary"

Planning Log

Every decision, action, user interaction, and tool check is logged to planning.log inside the roadmap directory.

Implementation Phase — /implement-roadmap v26 algorithm

Startup Sequence

  • Step 1: Resolve roadmap (coordinator scans ~/.roadmaps/ + Roadmaps/, filters by project field)
  • Step 1a: Check for stale artifacts (worktree, branch, PR, Implementing state) — prompt user immediately
  • Step 1b: Initialize implementation log
  • Step 1c: Mark as Implementing (only after cleanup)
  • Step 2: Start dashboard (auto-starts if not running)
  • Step 2b: Create feature branch and worktree (code only, no roadmap files)

Implementation Loop

  • 3a: Coordinator picks next step (deterministic, regex-based)
  • 3b: Print step info
  • 3c: Update dashboard (begin-step)
  • 3d: Launch worker agent via Agent tool
  • 3e: Verify completion, log commit, update dashboard (finish-step)
  • Repeat until done

Step 1: Create Draft PR

The first step pushes the branch and creates a draft PR titled Roadmap: <Name>. The PR stays draft throughout implementation.

Standard Steps: Implement, Build, Test, Push, Comment

Each step: implement code, build, test, update roadmap status in ~/.roadmaps/ (no git commit), push to branch, comment on the draft PR.

Final Step: Finalize & Merge PR

Populates Change History (commits table + PR link), copies <Name>-Roadmap.md flat file into repo, marks PR ready, waits for CI, reviews, merges, cleans up worktree.

Critical Invariant

The worktree is for CODE ONLY. Roadmap files stay in ~/.roadmaps/. The only roadmap artifact that enters the repo is the final flat file copy.

Worker Agent — implement-step-agent v11 algorithm

Implements exactly one step. Receives step number, description, complexity, roadmap path, worktree path, dashboard info, and log path in its prompt.

Three Flows

  • "Create Draft PR": Push branch, create draft PR, register PR on dashboard via pr-created
  • "Finalize & Merge PR": Populate Change History, copy flat roadmap file, mark PR ready, wait for CI, review, merge, cleanup
  • Standard: Read codebase, implement, build, test, update roadmap in ~/.roadmaps/, push, comment on PR

Rules

  • Implement ONLY the specified step
  • Never skip a step
  • Use git -C <path> instead of cd && git
  • Roadmap files live in ~/.roadmaps/, NOT in the worktree
  • On failure, log the error and return (don't retry silently)

04Coordinator Algorithm

The coordinator is a deterministic Python script. It reads roadmap files with regex, decides the next step, and outputs JSON. No LLM involved in step selection. [source]

next-step — Find the next implementable step algorithm

Iterates all ### Step N: sections in order. For each:

  • If status is Complete — skip
  • If type is Manual — add to manual_skipped, skip
  • Otherwise — this is the next step. Return it with the skipped manual steps.

If no implementable step remains, return {"action": "done"}.

for step in steps:
    if step.status == "Complete": continue
    if step.type == "Manual":
        manual_skipped.append(step)
        continue
    return {"action": "implement", "step": step, "manual_skipped": manual_skipped}
return {"action": "done", "manual_skipped": manual_skipped}

Key design decision: Manual steps are skipped, not blocked. The coordinator reports them so the skill can create GitHub issues and mark them on the dashboard.

resolve — Find and select a roadmap algorithm

Scans two locations:

  • ~/.roadmaps/<project>/ — working directory (drafts + in-progress)
  • Roadmaps/ in the repo — committed roadmaps

Project filtering: reads the project frontmatter field from each Roadmap.md. If it doesn't match the current git repo name, the roadmap is excluded.

Implementable check: only roadmaps in Ready or Implementing state are offered. Implementing is included so failed runs can be retried.

Returns single match as {"path": ...}, multiple as {"choose": [...]}, or {"error": ...}.

summary — Progress report algorithm

Parses all steps and returns counts:

{"feature": "AuthMiddleware", "total": 8, "complete": 5,
 "steps": [{"step": 1, "description": "...", "status": "Complete"}, ...]}

05State Machine

State is tracked via dated files in State/. The newest file (by date, then lifecycle order) determines the current state. [source]

Created → Planning → Ready → Implementing → Complete ↓ ↓ Declined Paused → Archived
Lifecycle ordering — Why not alphabetical? algorithm

State files are named YYYY-MM-DD-StateName.md. Sorting alphabetically puts Complete before Ready (C < R), which caused a bug where completed roadmaps showed as Ready.

Fix: sort by date first, then by lifecycle order:

_STATE_ORDER = {
    "Created": 0, "Planning": 1, "Ready": 2,
    "Implementing": 3, "Paused": 4,
    "Complete": 5, "Declined": 6, "Archived": 7,
}

Regression test: test_complete_beats_ready_on_same_date ensures Complete wins over Ready when both share the same date.

State predicates algorithm
  • is_active() — True for Created, Planning, Ready, Implementing, Paused
  • is_implementable() — True for Ready or Implementing (Implementing allows retrying failed runs)
  • current_state() — Returns the state name from the newest State/ file

Three-Source Scanning algorithm

The CLI scans three locations for each repo, deduplicating by roadmap ID:

  • Flat files first: Roadmaps/<Name>-Roadmap.md — finished roadmaps. Scanned first so completed versions win dedup.
  • Directories: Roadmaps/YYYY-MM-DD-<Name>/Roadmap.md — legacy layout. Skipped if flat file with same ID exists.
  • Working directory: ~/.roadmaps/<project>/ — drafts and in-progress roadmaps.

Regression test: test_flat_file_wins_over_implementing_directory_with_same_id — a completed flat file must beat a stale directory showing Implementing state.

Diagnose — Health Checks algorithm

roadmaps --diagnose checks for:

  • Orphaned directories: flat file exists with same ID (leftover from old Step 2c bug)
  • Stale Implementing: state file older than 24h (likely timed out)
  • Dangling worktrees: registered in git but directory missing
  • Missing project field: no project association in frontmatter
  • Missing ID: no dedup possible
